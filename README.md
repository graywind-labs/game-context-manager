# Game Context Manager

Game Context Manager is a local-first desktop GUI for creating, organizing, editing, and exporting game project context in files that ordinary agents can read.

游戏上下文管理器是一个本地优先的桌面 GUI 软件，用于创建、规整、编辑并导出普通 Agent 可直接读取的游戏项目上下文文件。

## Current MVP

The MVP can:

- Create or import a local `game-context/` workspace.
- Create and edit one game root node per workspace.
- Upload, rename, preview, link, unlink, and delete image assets.
- Create, edit, delete, and export module nodes and content nodes.
- Validate content `@image_id` / `@图片名` references against linked images.
- Generate and update `Markdown + YAML frontmatter`, `manifest.yml`, per-game `INDEX.md`, `image_catalog.yml`, and static downstream `AGENTS.md` / `CLAUDE.md` / `USAGE.md`.
- Configure a local OpenAI-compatible or `mock://` AI provider.
- Preview AI field edits and bottom-up summaries before the user confirms any overwrite.
- Rebuild the local SQLite index from an existing workspace generated by this app.

The example workspace in `examples/sample-workspace/` shows the agent-readable output shape.

## MVP Boundary

The app manages local game context and exports Markdown, YAML, static usage docs, indexes, and referenced image assets. It does not perform game analytics, image generation, RAG, MCP hosting, cloud collaboration, automatic optimization suggestions, or automatic gameplay.

本应用只管理本地游戏上下文，并导出 Markdown、YAML、静态使用说明、目录索引和被引用的图片资源。它不做游戏数据分析、生图、RAG、MCP 服务、云端协作、自动优化建议或自动玩游戏。

## Requirements

- Node.js with Corepack available.
- pnpm is pinned through `packageManager` as `pnpm@11.7.0`.
- Windows is the currently verified development environment.

Install dependencies:

```bash
corepack pnpm install
```

## Development

Run the desktop app in development mode:

```bash
corepack pnpm dev
```

Run TypeScript checks:

```bash
corepack pnpm typecheck
```

Run unit tests:

```bash
corepack pnpm test
```

Build the Electron/Vite output:

```bash
corepack pnpm build
```

Run the already-built app:

```bash
corepack pnpm start
```

`pnpm build` writes the production output to `out/`. This repository currently builds runnable Electron/Vite output, not a signed OS installer.

`pnpm lint` is not configured yet.

## Basic User Flow

1. Create or import a `game-context/` workspace.
2. Create or select a local user.
3. Create the game root node.
4. Upload and rename images.
5. Create module nodes and link images.
6. Create content nodes under modules and link images.
7. Use `@image_id` or `@图片名` in content only after linking the image to that content node.
8. Review generated Markdown previews and linked images in the GUI.
9. Optionally configure AI, generate a candidate edit or summary, inspect the diff, then confirm before overwrite.
10. Give external agents the generated `game-context/` folder.

## Agent Reading Flow

External agents should not need the GUI.

Recommended order:

1. Read `game-context/AGENTS.md` for Codex-style agents or `game-context/CLAUDE.md` for Claude Code / WorkBuddy.
2. Read `game-context/manifest.yml`.
3. Read `game-context/USAGE.md`.
4. Open the game path listed under `game.path`.
5. Use the per-game `INDEX.md` to choose relevant module and content files.
6. Resolve image references through `manifest.yml` or `games/<game_id>/image_catalog.yml`.

## Local Data

The GUI stores its editable index and settings in local SQLite:

```text
Electron app.getPath('userData')/game-context-manager.sqlite3
```

On the verified Windows development machine this resolves to:

```text
C:\Users\JT\AppData\Roaming\game-context-manager\game-context-manager.sqlite3
```

Generated agent-facing workspaces are stored wherever the user chooses when creating or importing a `game-context/` folder. API keys are stored only in the local SQLite database and are not written to exported Markdown, manifest, INDEX, image catalog, or sample workspace files.

## API And AI Safety

- API Base URL, API key, model name, and enabled state are configured in the GUI.
- `mock://local` can be used for tests and local smoke checks without a real API key.
- AI requests are only made after the user explicitly triggers an AI action.
- AI output is shown as a candidate with original text, new text, and diff; it is not written until the user confirms.
- Tests use mock providers and do not require real credentials.

## Verification

The current MVP checks are:

```bash
corepack pnpm typecheck
corepack pnpm test
corepack pnpm build
```

Manual smoke coverage should include launching the built app with `corepack pnpm start`, then creating or importing a workspace, creating a user, walking through game/image/module/content creation, checking Markdown previews, and confirming the generated files can be read without the GUI.

## Known Limits

- `pnpm lint` is not configured.
- SQLite uses Electron/Node `node:sqlite`, which currently emits an ExperimentalWarning during tests.
- The current workspace model is one game root node per workspace.
- Import is designed for workspaces generated by this app; complex hand-written Markdown conflicts are out of scope.
- There is no recycle bin, undo stack, cloud sync, multiplayer workspace, RAG, MCP server, or automatic gameplay.
