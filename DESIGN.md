# DESIGN.md

# Game Context Manager 详细设计

## 1. 产品定位

**Game Context Manager** 是一个本地优先的游戏上下文管理软件。

它的目标不是替代运营 Agent，而是为各种 Agent 提供稳定、模块化、可读取的游戏项目上下文。它负责把人类运营人员的粗输入、截图、主观标注和项目知识规整为标准文件结构，供 Codex、Claude Code、WorkBuddy、OpenClaw、Hermes 或其他 Agent 按目录读取。

核心定位：

```text
人类输入与维护游戏知识
        ↓
GUI 显示、编辑、规整、删除、汇总
        ↓
生成 agent 可用的 MD + manifest + 图片资源
        ↓
外部 Agent 在执行任务时读取上下文
```

## 2. 明确不做的事情

MVP 不做：

- 不做 RAG。
- 不做 MCP。
- 不做游戏数据分析。
- 不做生图。
- 不做自动提出运营优化建议的独立 Agent。
- 不做云端公共项目空间。
- 不做多人实时协作。
- 不做权限复杂的企业后台。
- 不做直接连接 TE、BI、广告后台。
- 不做自动玩游戏。

这些能力未来可以接入，但不属于当前软件的核心边界。

## 3. 核心需求摘要

软件需要承担：

1. 游戏上下文写入。
2. 游戏上下文规整。
3. 游戏上下文查看。
4. 游戏上下文编辑。
5. 游戏上下文删除。
6. 节点式文件管理。
7. 图片上传和重命名。
8. 图片与节点的多对多关联。
9. 在 GUI 中把图片和相关 MD 内容一起显示。
10. AI 辅助增加内容、修改内容、润色内容。
11. 子节点向父节点汇总。
12. 自动维护实时目录。
13. 生成静态通用使用说明。
14. 输出真实可被 Agent 使用的文件结构。
15. 支持本地账号和本地 API 配置。

## 4. 用户与角色

MVP 只需要本地账号，不需要联网注册。

### 4.1 本地用户

字段：

- 用户 ID
- 显示名
- 创建时间
- 最近登录时间

用途：

- 自动填充创建者。
- 自动填充最后编辑者。
- 记录 AI 辅助编辑仍由谁触发。

### 4.2 未来角色预留

未来可扩展：

- 管理员
- 编辑者
- 查看者
- 公共项目空间成员

MVP 不实现权限分层。

## 5. 工作空间与项目

### 5.1 工作空间

用户第一次创建游戏节点时，需要选择一个本地文件夹。软件会在该文件夹下创建：

```text
game-context/
```

并生成通用结构文件。

### 5.2 一个工作空间一个游戏主节点

MVP 中，一个工作空间只允许一个游戏主节点。

原因：

- 用户当前需求是围绕单个测试项目搭建上下文。
- 可以避免不同项目上下文混用。
- 后续如需多项目，可通过多个工作空间实现。

### 5.3 工作空间输出必须自洽

即使没有 GUI，Agent 或人也应能直接阅读 `game-context/` 文件夹理解项目上下文。

## 6. 节点模型

系统采用三级节点：

```text
一级：游戏主节点
二级：模块节点
三级：内容/阶段/体验节点
```

另有独立图片资产，不属于节点树，但可被多个节点关联。

## 7. 一级节点：游戏主节点

### 7.1 约束

- 每个工作空间只能有一个游戏主节点。
- 创建时必填结构字段。
- 可关联一张游戏 icon 或主界面图。
- 游戏主节点不能删除，除非删除整个工作空间。
- 游戏主节点字段修改会触发目录更新。

### 7.2 字段

| 字段 | 类型 | 必填 | 可编辑 | 说明 |
|---|---|---:|---:|---|
| 节点ID | string | 是 | 否 | 稳定 ID，用于文件名和引用 |
| 游戏名称 | string | 是 | 是 | 游戏展示名 |
| 游戏版本 | string | 是 | 是 | 当前项目版本 |
| 项目阶段 | enum | 是 | 是 | 立项/测试/上线/预备结项 |
| 游戏类型 | string | 否 | 是 | 如开箱like、SLG、模拟经营 |
| 核心玩法 | markdown/text | 否 | 是 | 游戏最核心的可重复循环 |
| 主要乐趣 | markdown/text | 否 | 是 | 主观体验总结 |
| 目标用户 | markdown/text | 否 | 是 | 目标玩家画像 |
| 当前运营目标 | markdown/text | 否 | 是 | 当前周期关注指标与目标 |
| 当前主要问题 | markdown/text | 否 | 是 | 当前项目主要问题 |
| 主要优化方向 | markdown/text | 否 | 是 | 当前方向性建议 |
| 创建者 | string | 是 | 否 | 登录账号自动填充 |
| 最后编辑者 | string | 是 | 否 | 每次编辑自动填充 |
| 更新时间 | datetime | 是 | 否 | 自动更新 |
| 补充说明 | markdown/text | 否 | 是 | 其他信息 |
| 主图ID | image_id | 否 | 是 | 可关联 icon 或主界面图 |

### 7.3 生成文件

```text
games/<game_id>/game.md
```

## 8. 二级节点：模块节点

### 8.1 定位

模块节点描述游戏中相对稳定的系统模块，例如：

- 开箱/生产线
- 战斗
- 养成
- 新手引导
- 首充
- 活动
- 签到
- 商店
- 任务系统

### 8.2 约束

- 一个游戏下可创建多个模块节点。
- 模块节点可删除。
- 删除模块节点时，其三级子节点需要一起删除或转移。MVP 采用“连同子节点一起删除，需要二次确认”。
- 节点 ID、所属游戏、创建者、创建时间不可编辑。
- 模块名称、模块定位修改会触发目录更新。
- 可关联零张或多张图片。

### 8.3 字段

| 字段 | 类型 | 必填 | 可编辑 | 说明 |
|---|---|---:|---:|---|
| 节点ID | string | 是 | 否 | 稳定 ID |
| 所属游戏 | game_id | 是 | 否 | 创建时自动填充 |
| 游戏版本 | string | 是 | 否 | 创建时版本快照 |
| 模块名称 | string | 是 | 是 | 显示名 |
| 模块定位 | markdown/text | 否 | 是 | 模块在游戏中的作用 |
| 系统规则 | markdown/text | 否 | 是 | 机制、规则、条件、限制 |
| 资源产出/消耗 | markdown/text | 否 | 是 | 产出、消耗、循环 |
| 关联截图 | image_id[] | 否 | 是 | 多对多关联 |
| 玩家主要操作 | markdown/text | 否 | 是 | 玩家实际操作 |
| 乐趣点（主观） | markdown/text | 否 | 是 | 主观体验 |
| 主要问题（主观） | markdown/text | 否 | 是 | 问题观察 |
| 优化方向（主观） | markdown/text | 否 | 是 | 方向性建议 |
| 创建者 | string | 是 | 否 | 自动填充 |
| 最后编辑者 | string | 是 | 否 | 自动填充 |
| 更新时间 | datetime | 是 | 否 | 自动更新 |

### 8.4 生成文件

```text
games/<game_id>/modules/<module_id>.md
```

## 9. 三级节点：内容/阶段/体验节点

### 9.1 定位

三级节点描述某个模块的具体部分、阶段、账号状态、付费层级、体验片段或流程。

例如：

- 新号第 1 天第一次进入开箱
- 第 3 天卡关后的礼包弹窗
- 累计付费 98 元账号的养成压力
- 某模块中的某个子系统
- 某个阶段的回流体验

### 9.2 约束

- 必须归属于一个模块。
- 可关联多张图片。
- 关联图片后，过程说明字段中允许用 `@image_id` 引用图片。
- 节点可删除。
- 删除节点不删除图片资产。
- 标题修改会触发目录更新。

### 9.3 字段

| 字段 | 类型 | 必填 | 可编辑 | 说明 |
|---|---|---:|---:|---|
| 节点ID | string | 是 | 否 | 稳定 ID |
| 所属游戏 | game_id | 是 | 否 | 自动填充 |
| 游戏版本 | string | 是 | 否 | 创建时版本快照 |
| 所属模块 | module_id | 是 | 否 | 创建时选择父节点 |
| 标题 | string | 是 | 是 | 显示名 |
| 关联截图 | image_id[] | 否 | 是 | 多对多关联 |
| 创角天数 | number/string | 否 | 是 | 第几天遇到 |
| 累计付费金额 | number/string | 否 | 是 | 账号付费档位 |
| 最大主线通关数 | string | 否 | 是 | 主线进度 |
| 角色等级 | string | 否 | 是 | 账号等级 |
| 过程说明 | markdown/text | 否 | 是 | 可用 `@image_id` 引用关联图片 |
| 乐趣点（主观） | markdown/text | 否 | 是 | 主观体验 |
| 已知问题（主观） | markdown/text | 否 | 是 | 问题观察 |
| 优化方向（主观） | markdown/text | 否 | 是 | 方向性建议 |
| 创建者 | string | 是 | 否 | 自动填充 |
| 最后编辑者 | string | 是 | 否 | 自动填充 |
| 更新时间 | datetime | 是 | 否 | 自动更新 |

### 9.4 生成文件

```text
games/<game_id>/contents/<content_id>.md
```

## 10. 图片资产模型

### 10.1 图片不是节点

图片独立管理，可被多个节点关联，也可以暂时不关联任何节点。

### 10.2 上传规则

上传图片时必须填写图片名。软件用该图片名生成新的安全文件名，覆盖原始乱码文件名。

保留原始文件名作为元数据。

### 10.3 字段

| 字段 | 类型 | 必填 | 可编辑 | 说明 |
|---|---|---:|---:|---|
| 图片ID | image_id | 是 | 否 | 稳定 ID |
| 图片名 | string | 是 | 是 | 用户填写的显示名 |
| 原始文件名 | string | 是 | 否 | 上传前文件名 |
| 文件路径 | string | 是 | 否 | 生成后的相对路径 |
| 文件类型 | string | 是 | 否 | png/jpg/webp 等 |
| 所属游戏 | game_id | 是 | 否 | 当前工作空间游戏 |
| 上传者 | string | 是 | 否 | 自动填充 |
| 更新时间 | datetime | 是 | 否 | 自动更新 |
| 备注 | text | 否 | 是 | 可选说明 |

### 10.4 多对多关联

关系表：

```text
node_image_links
- node_type: game | module | content
- node_id
- image_id
```

### 10.5 @ 引用规则

三级节点的过程说明中允许 `@image_id` 或 `@图片名`，但只有在该图片已与该节点关联时才合法。

如果用户输入了未关联图片引用：

- GUI 应提示“该图片未关联到此节点”。
- 可提供“一键关联并保留引用”。
- 不应静默生成无效引用。

## 11. AI 辅助能力

### 11.1 API 配置

GUI 提供配置页：

- API Base URL
- API Key
- 模型名
- 是否启用 AI 辅助
- 测试连接按钮

MVP 本地保存配置，不上传云端，不写入 agent-facing MD。

### 11.2 三种 AI 辅助编辑

#### A. 增加内容

输入：

- 原字段内容
- 用户新增的口语化内容
- 字段语义
- 节点上下文

要求：

- 保留原有内容。
- 将新增内容规整、合并到合适位置。
- 不删除原信息。
- 不编造新事实。

#### B. 修改内容

输入：

- 原字段内容
- 用户修改指令或替换内容
- 字段语义
- 节点上下文

要求：

- 按用户要求修改。
- 不改动无关事实。
- 如用户要求与原内容冲突，保留可追溯说明或在预览中突出差异。

#### C. 润色内容

输入：

- 原字段内容
- 字段语义
- 节点上下文

要求：

- 只改善表达、结构、术语统一。
- 不新增事实。
- 不改变判断强度。
- 不删除信息。

### 11.3 AI 输出必须预览

AI 输出不能直接覆盖字段。流程：

```text
点击 AI 辅助
↓
输入说明
↓
AI 生成候选版本
↓
展示原文 / 新文 / diff
↓
用户确认
↓
覆盖字段并更新最后编辑者
```

### 11.4 AI 规整层

创建节点时，人可以输入粗糙内容。AI 规整层可将粗输入转换为节点字段草案。

MVP 可先实现字段级 AI 编辑，后续再实现完整创建表单的“粗输入到字段草案”。

## 12. 自下而上总结功能

### 12.1 三级汇总到二级

根据某个模块下的三级内容节点，更新二级模块字段：

- 模块定位
- 系统规则
- 资源产出/消耗
- 玩家主要操作
- 乐趣点（主观）
- 主要问题（主观）
- 优化方向（主观）

原则：

- 主要补充父节点中没有的信息。
- 已有信息默认保留。
- 除非子节点中明确对应模块改动，否则不删除父节点已有内容。
- 输出为候选版本，用户确认后覆盖。

### 12.2 二级汇总到一级

根据所有二级模块节点，更新一级游戏节点字段：

- 核心玩法
- 主要乐趣
- 主要优化方向
- 当前主要问题

原则同上：

- 主要补充。
- 不轻易删改。
- 用户确认后覆盖。

### 12.3 汇总不是实时自动覆盖

汇总需要用户主动点击：

- “从子节点汇总到当前节点”
- “从模块汇总到游戏节点”

避免 AI 自动改变关键信息。

## 14. Manifest 设计

`manifest.yml` 是实时目录，随节点或关键字段变化自动更新。

包括：

- 工作空间信息
- 游戏主节点路径
- 模块节点列表
- 三级内容节点列表
- 图片资产列表
- 关系索引
- 任务上下文入口
- 生成时间

示例：

```yaml
workspace:
  schema_version: 1
  generated_at: 2026-06-18T12:00:00+09:00

game:
  node_id: mission_frontline
  name: 使命防线
  version: v0.3.2
  path: games/mission_frontline/game.md
  index: games/mission_frontline/INDEX.md

modules:
  gacha_workshop:
    name: 生产线/开箱
    path: games/mission_frontline/modules/gacha_workshop.md
    images:
      - img_main_workshop

contents:
  day1_first_gacha:
    title: 第一天首次进入生产线
    module_id: gacha_workshop
    path: games/mission_frontline/contents/day1_first_gacha.md
    images:
      - img_main_workshop
      - img_gacha_result

images:
  img_main_workshop:
    name: 主界面生产线区域
    path: games/mission_frontline/assets/images/img_main_workshop__main-workshop.png
    linked_nodes:
      - game:mission_frontline
      - module:gacha_workshop
      - content:day1_first_gacha
```

## 15. 使用说明与目录解耦

### 15.1 静态通用使用说明

`USAGE.md`、下游 `AGENTS.md`、下游 `CLAUDE.md` 定义的是通用工作流：

- 如何读取 `manifest.yml`
- 如何识别游戏上下文
- 如何根据任务读取模块
- 如何处理图片引用
- 如何尊重用户手动 @ 的上下文

它们不随每次节点变化重写。

### 15.2 实时目录

以下文件随节点变化自动更新：

- `manifest.yml`
- `games/<game_id>/INDEX.md`
- `games/<game_id>/image_catalog.yml`

## 16. GUI 设计

### 16.1 整体风格

目标：像 OpenWebUI 一样简洁、现代、可持续使用；节点管理像 XMind 一样清晰。

布局建议：

```text
顶部栏：工作空间 / 当前用户 / API 设置 / 导出状态
左侧栏：节点树 + 图片库入口
中间区：当前节点详情与编辑
右侧栏：关联截图、MD 预览、AI 辅助编辑、历史信息
底部/通知：保存状态、目录更新状态、错误提示
```

### 16.2 节点树

树结构：

```text
游戏主节点
  模块节点 A
    内容节点 A1
    内容节点 A2
  模块节点 B
    内容节点 B1
```

功能：

- 创建子节点
- 选择节点
- 删除二/三级节点
- 展开/折叠
- 搜索节点标题
- 显示节点类型图标
- 显示缺字段提示

### 16.3 节点详情页

应把 MD 文件和关联图片整合显示：

- 基本信息表单
- 可编辑字段
- 不可编辑结构字段
- 关联图片列表
- 过程说明中的 @ 图片引用提示
- 保存按钮
- AI 辅助按钮
- MD 预览

### 16.4 图片库

功能：

- 上传图片
- 必填图片名
- 自动改写文件名
- 查看图片
- 编辑图片名/备注
- 查看被哪些节点引用
- 删除图片
- 删除图片前提示哪些节点正在引用

### 16.5 AI 辅助面板

模式：

- 增加内容
- 修改内容
- 润色
- 子节点汇总到父节点

需要展示：

- 原内容
- 用户输入
- AI 输出
- 差异
- 确认覆盖按钮
- 取消按钮

## 17. 删除规则

### 17.1 删除三级节点

- 删除其 MD 文件。
- 删除其数据库记录。
- 删除其节点图片关联。
- 不删除图片资产。
- 更新 manifest 和 INDEX。

### 17.2 删除二级节点

- 二次确认。
- 删除该模块 MD。
- 删除所有子内容 MD。
- 删除相关节点图片关联。
- 不删除图片资产。
- 更新 manifest 和 INDEX。

### 17.3 删除图片

- 如果未被任何节点引用，可直接删除。
- 如果被引用，需要显示引用列表并二次确认。
- 删除后从所有节点关联中移除。
- 更新 manifest、image_catalog 和相关 MD。

## 18. 数据存储策略

### 18.1 推荐策略

MVP 采用：

```text
SQLite 作为 GUI 编辑状态和索引
MD/YAML/图片作为 agent-facing 输出
```

SQLite 存：

- 用户
- 工作空间路径
- 节点记录
- 图片记录
- 关联关系
- API 配置
- 编辑历史摘要
- AI 编辑候选记录

文件系统存：

- Agent 可读 MD
- 图片
- manifest
- INDEX
- 使用说明

### 18.2 恢复与重建

后续可增加“从 game-context 文件夹重建 SQLite 索引”的功能。MVP 可以在后期任务实现。

## 19. API 配置与安全

- API key 只存本地配置。
- API key 不写入 MD。
- AI 请求只在用户点击 AI 功能时发起。
- AI 请求内容应只包含必要节点字段与必要上下文。
- 测试必须使用 mock provider，不需要真实 key。

## 20. 边界条件与盲区补充

### 20.1 版本字段

- 游戏主节点的游戏版本表示当前项目版本，可编辑。
- 二/三级节点的游戏版本表示创建时的版本快照，默认不可编辑。
- 如需迁移到新版本，后续可以做“复制节点到新版本”。

### 20.2 ID 生成

节点 ID 应使用安全 slug：

- 小写英文、数字、下划线、短横线。
- 中文名称需要自动转拼音或用时间/hash 生成。
- 用户可在创建时手动填写 ID。
- ID 一旦创建不可编辑。

### 20.3 字段空缺

创建时不要求填满所有内容字段。空字段在 MD 中仍保留标题，以提醒后续补充。

### 20.4 主观字段

带“主观”的字段必须保留主观性，不应被 AI 改写成确定事实。

### 20.5 AI 不应替人下结论

AI 规整应帮助表达和归类，不应自行制造游戏事实。

### 20.6 多 Agent 兼容

生成文件必须满足：

- 人能直接读。
- Codex 能按 `AGENTS.md` 和 `manifest.yml` 读。
- Claude Code/WorkBuddy 能按 `CLAUDE.md` 和目录读。
- 不支持特殊工具的 Agent 也可通过复制 MD 内容使用。

## 21. MVP 验收标准

MVP 完成时至少能：

1. 本地注册/选择用户。
2. 选择文件夹创建一个 game-context 工作空间。
3. 创建一个游戏主节点。
4. 上传图片并重命名。
5. 创建模块节点并关联图片。
6. 创建三级内容节点并关联多图。
7. 在过程说明中引用已关联图片。
8. 查看节点时同时显示字段和关联图片。
9. 手动编辑字段并写入 MD。
10. 删除二/三级节点并更新目录。
11. 自动生成并更新 manifest、INDEX、image_catalog。
12. 生成静态 USAGE、AGENTS、CLAUDE 文件。
13. 配置 API URL/key。
14. 用 mock 或真实 API 执行 AI 增加、修改、润色的预览式编辑。
15. 执行三级到二级、二级到一级的汇总预览。
16. 通过至少一个示例工作空间验证 Agent 可读取文件。
