# DESIGN.md

# Game Context Manager 详细设计

## 1. 产品定位

**Game Context Manager（游戏上下文管理器）** 是一个本地优先的桌面 GUI 软件，用来按节点管理给人类和 Agent 双方阅读的游戏上下文，并在节点详情中提供 AI 辅助增加、修改、润色和汇总能力。

核心链路：

```text
本地用户登录
  ↓
创建或导入带唯一标记的本地工作区
  ↓
按 游戏主节点 → 模块节点 → 内容节点 管理上下文
  ↓
在节点详情中编辑字段、关联图片、预览 MD、调用 AI
  ↓
用户主动导出 AGENTS/CLAUDE 与目录索引文件
  ↓
外部 Agent 读取 Markdown + YAML frontmatter + 图片资源
```

GUI 是面向人的管理层；导出的普通文件是面向 Agent 的上下文源。

## 2. 明确不做的事情

MVP 和本轮重构不做：RAG、MCP、游戏数据分析、生图、自动提出运营优化建议的独立 Agent、云端公共项目空间、多人实时协作、复杂企业权限、连接 TE/BI/广告后台、自动玩游戏、AI 未确认直接覆盖用户内容。

## 3. 核心需求摘要

软件需要承担：

1. 本地用户登录与最近用户恢复。
2. 工作区创建、唯一标记写入、已有工作区导入。
3. 游戏主节点、模块节点、内容节点的分级树管理。
4. 节点详情查看、编辑、保存、删除。
5. 图片上传、重命名、图库管理、节点关联。
6. 在节点详情中显示关联图片、MD 预览与 AI 辅助入口。
7. AI 辅助增加内容、修改内容、润色内容。
8. 子节点向父节点汇总。
9. 手动导出 `AGENTS.md` / `CLAUDE.md`。
10. 手动生成或更新 `manifest.yml`、单游戏 `INDEX.md`、`image_catalog.yml`。
11. 输出真实可被 Agent 直接读取的文件结构。
12. 支持中文和英文界面，且同一时刻 UI 文案只能使用一种语言。
13. 支持本地 API 配置；AI 默认可用，但在 API 未配置或连接失败时提示不可用。

不再保留旧版三栏工作台、右侧常驻辅助面板、顶部语言切换按钮、主界面 API 配置区、AI 启用/禁用复选框、实时自动目录更新、工作空间概览、快捷操作、推进路径等冗余 UI。

## 4. 用户与登录

MVP 只需要本地账号，不需要联网注册、不需要密码。

本地用户字段：用户 ID、显示名、创建时间、最近登录时间。

应用启动时：

- 如果没有本地用户，或没有可恢复的最近登录用户，必须先弹出登录/创建用户弹窗。
- 弹窗中只要求填写本地显示名。
- 未完成本地登录前，软件主体不可操作。
- 退出当前账号后，关闭设置界面或回到主界面时，应再次弹出登录弹窗，直到用户完成登录。

当前用户用于自动填充创建者、最后编辑者，并用于删除权限判断。删除节点或图片时，当前用户必须与节点创建者或图片上传者一致。

## 5. 工作区与导入/创建

### 5.1 工作区唯一标记

本软件创建的上下文工作区必须包含唯一标记文件，建议命名为：

```text
.game-context-manager.yml
```

标记文件至少包含 schema version、workspace id、generator/app id、创建时间、主节点 ID、主节点文件夹名。

导入时，用户点击左上角“导入”按钮并选择一个文件夹。软件必须扫描所选文件夹中的工作区标记：

- 读取到且只读取到一个合法标记：允许导入。
- 没有读取到合法标记：提示未识别到游戏上下文，导入失败。
- 读取到多个合法标记：提示该文件夹包含多个工作区标记，导入失败。

导入只面向本软件生成的工作区。复杂冲突合并、手写非标准 Markdown、多工作区合并不属于当前范围。

### 5.2 创建工作区与主节点

如果尚未打开任何工作区：

- 左侧节点树显示灰色占位层级：创建主节点、创建模块节点、创建内容节点。
- 右侧详情区不显示节点表单，只提示“请选择一个节点来查看”或对应语言文案。
- 左上角提供“导入”按钮。
- 灰色主节点右侧提供 `+`，用于创建主节点，也就是创建新的工作区。

创建主节点时弹出应用内弹窗。弹窗只显示创建主节点必须填写的信息，选填字段不在创建弹窗中出现。创建弹窗必须包含游戏名称、游戏版本、项目阶段、选择文件位置按钮。

每次创建主节点都必须重新选择一个本地文件夹，不能默认复用上一次打开或创建过的工作区路径。用户选择的文件夹就是工作区根目录，Agent 应在该根目录下运行；`AGENTS.md`、`CLAUDE.md` 和 `manifest.yml` 与主节点文件夹同级。软件在所选根目录内创建一个以“主节点名称 + 游戏上下文”清洗后得到的子文件夹，作为主节点游戏上下文文件夹。节点 ID 由程序自动分配，不允许用户手动填写。取消选择或创建失败时，若当前已经打开工作区，必须保留当前工作区不变。

创建成功后自动选中主节点，右侧显示主节点详情，左侧从灰色占位切换为真实文件/节点结构。

### 5.3 一个工作区一个游戏主节点

MVP 中，一个工作区只允许一个游戏主节点。后续如需多项目，可通过多个工作区实现。

## 6. 生成的文件结构

目标生成结构：

```text
<selected-folder>/
  .game-context-manager.yml          # 工作区唯一标记，导入时必须识别且只能有一个
  AGENTS.md                          # 手动导出的 Codex/通用 Agent 说明
  CLAUDE.md                          # 手动导出的 Claude Code / WorkBuddy 说明
  manifest.yml                       # 手动生成/更新的机器可读目录

  <game_folder_name>/                 # 以“主节点名称 + 游戏上下文”清洗后生成，例如：使命防线游戏上下文
    game.md                           # 一级游戏主节点
    INDEX.md                          # 手动生成/更新的单游戏目录索引
    image_catalog.yml                 # 手动生成/更新的图片元数据目录
    assets/
      images/
        <image_id>__<display_name>.<ext>
    modules/
      <module_id>/
        module.md                     # 二级模块节点
        contents/
          <content_id>.md             # 三级内容/阶段/体验节点
```

说明：

- 左侧显示的节点、图片库和索引文件，应尽量与实际文件层级保持一致。
- 图片库在 UI 中与模块节点同级显示；实际图片文件仍存放在 `assets/images/`。
- `AGENTS.md` 和 `CLAUDE.md` 与主节点文件夹同级，不放入主节点文件夹。
- `manifest.yml`、`INDEX.md`、`image_catalog.yml` 不再因为节点或图片变化而实时自动重写，而是在用户点击“导出当前目录”后生成或更新。
- 节点对应的 Markdown 文件仍应在节点保存时写入，以保证用户内容不只存在数据库中。
- 管理器内的新增、修改、删除应尽量落实到真实文件；点击主界面右上角刷新按钮时，应重新扫描当前工作区磁盘文件并用扫描结果刷新左侧目录和本地数据库快照。应用重启后应自动恢复最近工作区，并以磁盘文件重新扫描结果为准。

旧版 `game-context/` 外层目录结构属于 T020 之前的实现形态。后续任务需要迁移服务、测试和示例工作区到本节结构。

## 7. 节点模型与 ID

系统采用三级节点：一级游戏主节点、二级模块节点、三级内容/阶段/体验节点。另有独立图片资产和目录索引文件。图片不是节点，但会在左侧树中以橙色图片条目显示。

所有节点 ID 和图片 ID 都是必要信息，但不由用户手动填写。程序根据层级和创建顺序自动分配简单、稳定、容易识别的 ID，例如：`game_001`、`module_001`、`content_001`、`image_001`。同一工作区内 ID 必须唯一。ID 一旦创建不可编辑，并用于文件路径、frontmatter、关联关系和 `@image_id` 引用。

## 8. 一级节点：游戏主节点

约束：

- 每个工作区只能有一个游戏主节点。
- 创建时只填写必填字段；选填字段在详情页补充。
- 创建节点弹窗中，用户按空格键应等同点击创建按钮；但焦点位于输入框、文本域、下拉框或按钮内时，空格保留原生输入/控件行为。
- 可关联一张游戏 icon 或主界面图。
- 节点 ID、创建者、创建时间不可编辑。
- 当前用户必须是创建者时才允许删除主节点；删除主节点需要二次确认并说明会删除该主节点的整个游戏上下文文件夹，包括模块、内容和图片资源文件。

字段：

| 字段 | 类型 | 必填 | 可编辑 | 说明 |
|---|---|---:|---:|---|
| 节点ID | string | 是 | 否 | 程序自动分配 |
| 游戏名称 | string | 是 | 是 | 游戏展示名 |
| 游戏版本 | string | 是 | 是 | 当前项目版本 |
| 项目阶段 | enum | 是 | 是 | 立项/测试/上线/预备结项 |
| 游戏类型 | string | 否 | 是 | 如开箱like、SLG、模拟经营 |
| 核心玩法 | markdown/text | 否 | 是 | 游戏最核心的可重复循环 |
| 主要乐趣 | markdown/text | 否 | 是 | 主观体验总结 |
| 目标用户 | markdown/text | 否 | 是 | 目标玩家画像 |
| 当前运营目标 | markdown/text | 否 | 是 | 当前周期关注指标与目标 |
| 当前主要问题 | markdown/text | 否 | 是 | 当前项目主要问题 |
| 主要优化方向 | markdown/text | 否 | 是 | 当前方向性建议 |
| 创建者 | string | 是 | 否 | 登录账号自动填充 |
| 最后编辑者 | string | 是 | 否 | 每次编辑自动填充 |
| 更新时间 | datetime | 是 | 否 | 自动更新 |
| 主图ID | image_id | 否 | 是 | 从图片库下拉选择 |

详情页要求：

- 右侧只显示当前选中主节点的详情。
- 保存按钮只有在内容发生修改后才可点击，否则为禁用状态。
- 主图选择使用下拉框，下拉项包含图片名和预览图。
- 下拉项提供“详情”入口，可查看原图。
- 在节点 ID 和名称等基础信息下方提供“AI辅助编辑”分组框，框内按钮为“增加”“修改”“润色”“汇总”；汇总按钮与其他 AI 按钮一样使用填充色。
- MD 预览通过详情页按钮打开弹窗显示，不在详情页底部内嵌预览栏。

生成文件：`<game_folder_name>/game.md`。

## 9. 二级节点：模块节点

模块节点描述游戏中相对稳定的系统模块，例如开箱/生产线、战斗、养成、新手引导、首充、活动、签到、商店、任务系统。

创建入口：

- 未创建模块时，左侧灰色“创建模块节点”右侧 `+` 可创建模块。
- 主节点已存在后，主节点右侧 `+` 也用于创建模块。
- 创建模块时弹出应用内弹窗，只填写创建所需必填信息。
- 模块 ID 自动分配，用户不可手填。
- 创建成功后自动选中新模块，并在右侧显示模块详情。

字段：

| 字段 | 类型 | 必填 | 可编辑 | 说明 |
|---|---|---:|---:|---|
| 节点ID | string | 是 | 否 | 程序自动分配 |
| 所属游戏 | game_id | 是 | 否 | 创建时自动填充 |
| 游戏版本 | string | 是 | 否 | 创建时版本快照 |
| 模块名称 | string | 是 | 是 | 显示名 |
| 模块定位 | markdown/text | 否 | 是 | 模块在游戏中的作用 |
| 系统规则 | markdown/text | 否 | 是 | 机制、规则、条件、限制 |
| 资源产出/消耗 | markdown/text | 否 | 是 | 产出、消耗、循环 |
| 关联截图 | image_id[] | 否 | 是 | 多对多关联 |
| 玩家主要操作 | markdown/text | 否 | 是 | 玩家实际操作 |
| 乐趣点 | markdown/text | 否 | 是 | 主观体验 |
| 主要问题 | markdown/text | 否 | 是 | 问题观察 |
| 优化方向 | markdown/text | 否 | 是 | 方向性建议 |
| 创建者 | string | 是 | 否 | 自动填充 |
| 最后编辑者 | string | 是 | 否 | 自动填充 |
| 更新时间 | datetime | 是 | 否 | 自动更新 |

详情页要求：保存按钮仅在内容修改后启用；关联图片区域显示已关联图片名称和预览图，并位于“AI辅助编辑”分组框下方、模块定位上方；多个已关联图片横向排列，超出当前宽度时只在“关联截图”栏位内部出现横向滚动条并通过该滚动条拖拽查看，不得撑出详情页边框或触发主界面横向滚动；点击图片可查看大图；图片库为空时显示“暂无图片”；区域最右侧提供 `+`，用于弹出当前所有未关联图片；在节点 ID 和名称等基础信息下方提供“AI辅助编辑”分组框，框内按钮为“增加”“修改”“润色”“汇总”。

生成文件：`<game_folder_name>/modules/<module_id>/module.md`。

## 10. 三级节点：内容/阶段/体验节点

三级节点描述某个模块的具体部分、阶段、账号状态、付费层级、体验片段或流程。

创建入口：

- 未创建内容时，模块下灰色“创建内容节点”右侧 `+` 可创建内容。
- 模块已有内容后，模块节点右侧 `+` 用于继续创建该模块下的新内容。
- 创建内容时弹出应用内弹窗，只填写创建所需必填信息。
- 内容 ID 自动分配，用户不可手填。
- 创建成功后自动选中新内容，并在右侧显示内容详情。

字段：

| 字段 | 类型 | 必填 | 可编辑 | 说明 |
|---|---|---:|---:|---|
| 节点ID | string | 是 | 否 | 程序自动分配 |
| 所属游戏 | game_id | 是 | 否 | 自动填充 |
| 游戏版本 | string | 是 | 否 | 创建时版本快照 |
| 所属模块 | module_id | 是 | 否 | 创建时选择父节点 |
| 标题 | string | 是 | 是 | 显示名 |
| 关联截图 | image_id[] | 否 | 是 | 多对多关联 |
| 创角天数 | number/string | 否 | 是 | 第几天遇到 |
| 累计付费金额 | number/string | 否 | 是 | 账号付费档位 |
| 最大主线通关数 | string | 否 | 是 | 主线进度 |
| 角色等级 | string | 否 | 是 | 账号等级 |
| 过程说明 | markdown/text | 否 | 是 | 可用 `@image_id` 引用关联图片 |
| 乐趣点 | markdown/text | 否 | 是 | 主观体验 |
| 已知问题 | markdown/text | 否 | 是 | 问题观察 |
| 优化方向 | markdown/text | 否 | 是 | 方向性建议 |
| 创建者 | string | 是 | 否 | 自动填充 |
| 最后编辑者 | string | 是 | 否 | 自动填充 |
| 更新时间 | datetime | 是 | 否 | 自动更新 |

详情页逻辑与模块一致；关联截图的横向滚动条只属于“关联截图”栏位内部，不得触发主界面横向滚动。过程说明中的 `@image_id` 或 `@图片名` 必须来自该内容节点已关联图片。

生成文件：`<game_folder_name>/modules/<module_id>/contents/<content_id>.md`。

## 11. 图片资产与图片库

图片独立管理，可被多个节点关联，也可以暂时不关联任何节点。

左侧树中，在与模块节点同级的位置显示“图片库”栏位：节点圆圈使用当前青色，图片库和图片条目使用橙色；图片库栏位不显示图片数量；图片库右侧 `+` 打开上传图片弹窗；图片库下方显示当前所有已上传图片的图片名和预览小图；点击图片条目后，右侧显示图片详情和原图。

上传图片必须进入上传弹窗，填写必要信息并选择图片。触发方式包括：点击图片库 `+`、将图片拖拽到主界面左侧、在上传弹窗的选择图片框内拖拽图片、在左侧栏右键粘贴或按 `Ctrl+V` 粘贴剪贴板图片。上传弹窗中，用户按空格键应等同点击上传按钮；但焦点位于输入框、文本域、下拉框或按钮内时，空格保留原生输入/控件行为。

上传规则：必须填写图片名；根据图片名生成安全文件名；界面不显示原始文件名或生成路径；除非用户明确触发相关 AI 功能且任务允许，否则 API 或 AI 功能不得自动接收图片。

图片详情页显示图片 ID、图片名、预览图、备注和关联节点列表。关联节点列表用于反查某张图片被哪些游戏、模块或内容节点引用。

三级节点的过程说明允许 `@image_id` 或 `@图片名`，但只有在该图片已与该节点关联时才合法。非法引用必须提示，不应静默生成无效引用。

## 12. 主界面信息架构

主界面只有左右两部分：

```text
左侧：工作区导入、节点/文件树、图片库、导出按钮、设置入口
右侧：当前选中节点、图片或索引文件的详情与操作
```

要求：

- 不再保留顶部状态栏作为主要信息承载区。
- 不再保留右侧常驻辅助面板。
- 不再平铺工作空间、本地用户、图片、模块、内容、API 等所有功能。
- 选中一个节点时，右侧只显示该节点详情，不显示其他节点或图片库内容。
- 未选中节点时，右侧只显示“请选择一个节点来查看”的空状态。
- 右侧详情页应随主窗口宽度自适应拉宽，最大化后文本填充框、图片关联栏位和预览区域应使用更多可用空间，不得被固定窄版最大宽度限制。
- 风格和配色可以沿用当前视觉方向，但信息结构必须改为节点树 + 详情页。

左上角提供“导入”按钮。点击导入时，如果当前已经打开工作区，界面不得立刻退出当前工作区；只有用户实际选择文件夹且导入成功后才切换到新工作区，取消选择或导入失败时必须保留当前工作区。左侧节点树显示游戏主节点、模块节点、内容节点、图片库、手动生成的索引文件。已创建的主节点、模块节点、图片库、Agent 文件分组和索引文件分组应提供折叠/展开按钮；没有子项时折叠按钮禁用。创建子节点的 `+` 按钮应使用青色填充白色图标，与折叠后的 `+` 按钮做出视觉区分。左下角提供“导出当前 AGENTS.md / CLAUDE.md”、“导出当前目录”和设置图标；这些左下角操作必须固定在主界面左下角，不随右侧详情页滚动或左侧节点树滚动而移出视野。

应用标题、左上角产品名和窗口标题必须随语言设置切换：默认中文显示“游戏上下文管理器”，英文显示“Game Context Manager”。Electron 默认的 File/Edit/View/Window/Help 菜单不属于当前功能范围，应隐藏。

右侧详情区根据左侧选中项切换为游戏详情、模块详情、内容详情、图片详情、Agent 文件详情、目录索引文件详情或空状态。Agent 文件和目录索引文件详情需要显示文件内容预览，不显示文件路径栏位。所有详情页都应使用明确的保存状态：内容未修改时保存按钮禁用，内容修改后保存按钮启用，保存成功后回到禁用状态。节点保存成功或失败都必须显示短暂反馈提示。节点详情底部的程序生成信息保留为无标题信息框；创建者和最后编辑者显示本地用户名，不直接显示内部用户 ID。

MD 预览不再放在常驻辅助面板中。每个节点详情页提供“MD 预览”按钮，点击后通过弹窗显示 Markdown 内容。

## 13. 设置

设置入口位于左下角。设置中包含：退出当前账号、软件语言选择、API 配置、帮助入口。帮助按钮位于 API 配置下方、当前用户上方；点击后以与 Agent 文件和索引文件预览一致的弹窗方式显示应用根目录 `README.md`。

设置中不包含主界面语言切换按钮，也不包含“启用 AI 辅助”复选框。

语言规则：UI 文案不可中英文并列；打开应用默认中文；选择中文时全界面和窗口标题使用中文；选择英文时全界面和窗口标题使用英文；新增文案必须进入统一文案表或等价 i18n 结构。

API 配置包含 API Base URL、API Key、模型名、测试连接按钮。AI 功能默认视为启用。若 API 未配置、配置不完整或连接失败，用户点击 AI 功能时提示 API 不可用。

## 14. AI 辅助能力

每个节点详情页在节点 ID 和名称等基础信息下方提供“AI辅助编辑”分组框。分组框中提供不同颜色的按钮：增加、修改、润色；模块和游戏详情还提供“汇总”按钮。点击后打开 AI 弹窗。

AI 弹窗上半部分包含字段多选下拉框、已选字段 tag、用户指令输入框、生成按钮。每个 tag 显示字段名和 `x` 删除按钮。

生成后，弹窗下半部分展开结果区域，而不是打开第二个弹窗或覆盖上半部分。结果区域只显示被选择字段的原内容、新内容和差异；AI 生成的新内容解析为结构化字段数据；用户可以编辑 AI 生成的新内容、取消、确认覆盖，或修改上方字段选择/指令后再次生成。

AI 修改模式必须在 prompt 和解析逻辑中约束模型只修改用户选择的字段。

汇总按钮融入节点详情：模块节点和游戏主节点都在“AI辅助编辑”分组框中提供“汇总”按钮。点击后打开与 AI 修改相同逻辑的弹窗，展示原内容、新内容和差异，用户可编辑、取消或确认覆盖。

AI 输出不能直接覆盖字段。API key 只存本地配置，不写入 Markdown、YAML、索引或示例文件。AI 请求只在用户点击 AI 功能时发起。测试必须使用 mock provider。

## 15. 文件输出与目录索引

每个节点 MD 必须包含 YAML frontmatter，保证 Agent 和程序都能读取结构信息。

`manifest.yml`、单游戏 `INDEX.md`、`image_catalog.yml` 是目录索引文件。它们不再随节点、图片或关键字段变化实时自动更新。用户需要点击左下角“导出当前目录”，软件才根据当前文件和数据库状态生成或修改这些索引文件。

导出当前目录后：`manifest.yml` 写在所选工作区文件夹下，与主节点文件夹同级；`INDEX.md` 写在主节点文件夹下；`image_catalog.yml` 写在主节点文件夹下；左侧树显示这些索引文件，且与实际文件层级一致。

导出的 Agent 文件和目录索引文件必须跟随设置中的界面语言。中文设置下，AGENTS/CLAUDE 正文、INDEX 人类可读标题与说明、manifest/image_catalog 中可读描述应使用中文；英文设置下使用英文。文件名、路径、YAML 机器字段名和枚举原始值等需要稳定解析的字段保持英文。

用户点击左下角“导出当前 AGENTS.md / CLAUDE.md”后，在当前所选工作区文件夹下生成或更新 `AGENTS.md` 和 `CLAUDE.md`。这两个文件与主节点文件夹同级，不在主节点文件夹内部。导出成功或失败提示应为短暂提示，约 2 秒后自动消失。

`AGENTS.md`、`CLAUDE.md` 是静态工作流说明。除非模板被主动更新或用户主动导出，否则不应因节点增删改而每次重写。

`AGENTS.md` 面向 Codex/通用 Agent，`CLAUDE.md` 面向 Claude Code / WorkBuddy 等 Claude 系 Agent。两者必须是同等完整的任务手册，只在首段身份说明和必要措辞上适配不同 Agent，不应让任一类 Agent 缺少关键规则。两者应指导下游 Agent 如何按任务选择上下文：默认读取游戏主节点；内容节点任务需要同时读取父模块；图片/UI 任务需要通过 `image_catalog.yml` 反查图片关联节点；维度型问题应优先横向查阅三层节点的对应结构化字段；竞品、市场、用户分析应先从主节点到模块把握游戏全貌，再在需要且环境允许时查询外部资料；回答中应说明依据、推测依据或明确承认上下文不足。

`manifest.yml` 应包含机器可读的 `field_schema`，说明游戏主节点、模块节点、内容节点分别有哪些结构化字段、字段位置和适用场景。`INDEX.md` 应包含同一字段地图的人类可读版本，便于 Agent 在横向维度问题中先定位字段再读取节点全文。

`image_catalog.yml` 在保留 `linked_nodes` 的同时，应尽量写入 `linked_node_files`，列出图片关联节点的类型、ID、名称和路径。用户 @ 图片或围绕图片提问时，下游 Agent 可以直接从该字段反查需要读取的节点文件。

Agent 文件和目录索引文件详情页右上角提供“删除”按钮。删除只移除对应已知导出文件本身；之后用户可在需要时通过左下角手动导出重新生成。

## 16. 右键菜单与系统操作

左侧栏支持右键菜单：

- 粘贴：如果剪贴板中有图片，打开上传图片弹窗并预填图片。
- 在文件夹中打开：根据当前右键选中的节点、图片或索引文件，在系统文件管理器中打开对应位置并选中对应文件。
- 删除：当鼠标位于文件、图片或节点条目上右键时，菜单最下方显示删除选项；点击后进入与该条目详情页删除按钮相同的确认弹窗和删除流程。

打开文件夹需要通过 Electron main process 调用系统能力。

## 17. 删除规则

删除三级节点：必须二次确认；当前用户必须是创建者；删除其 MD 文件、数据库记录和节点图片关联；不删除图片资产；将目录索引标记为需要重新导出，或提示索引可能过期。

删除二级节点：必须二次确认；当前用户必须是创建者；删除该模块 MD、所有子内容 MD 和相关节点图片关联；不删除图片资产；将目录索引标记为需要重新导出，或提示索引可能过期。

删除游戏主节点：必须二次确认，并清楚提示会影响该主节点下的模块、内容和图片关联；当前用户必须是创建者；不得在没有明确确认的情况下删除用户文件夹外的内容。MVP 可先实现删除主节点记录与主节点文件，是否级联删除整个主节点文件夹需要在任务中谨慎处理。

删除图片：当前用户必须是上传者；如果被引用，需要显示引用列表并二次确认；删除后从所有节点关联中移除；删除图片不应删除节点；将目录索引标记为需要重新导出，或提示索引可能过期。

删除 Agent 文件或目录索引文件：必须二次确认；只允许删除当前工作区内程序识别的 `AGENTS.md`、`CLAUDE.md`、`manifest.yml`、`INDEX.md`、`image_catalog.yml`，不得接受任意路径删除。删除后左侧文件列表和右侧详情应同步刷新。

## 18. 数据存储策略

MVP 采用 SQLite 作为 GUI 编辑状态和索引，MD/YAML/图片作为 agent-facing 输出。

SQLite 存：用户、工作区路径和唯一标记信息、节点记录、图片记录、关联关系、API 配置、编辑历史摘要、AI 编辑候选记录、UI 语言偏好、最近登录用户。

文件系统存：工作区唯一标记、Agent 可读 MD、图片、手动导出的 manifest、手动导出的 INDEX、手动导出的 image_catalog、手动导出的 AGENTS/CLAUDE。

## 19. 边界条件

- 游戏主节点的游戏版本表示当前项目版本，可编辑；二/三级节点的游戏版本表示创建时的版本快照，默认不可编辑。
- 创建时不要求填满所有内容字段。空字段在 MD 中仍保留标题，以提醒后续补充。
- 乐趣点、问题、优化方向等体验判断字段必须保留主观性，不应被 AI 改写成确定事实。
- AI 规整应帮助表达和归类，不应自行制造游戏事实。
- 生成文件必须让人能直接读，Codex 能按 `AGENTS.md` 和 `manifest.yml` 读，Claude Code/WorkBuddy 能按 `CLAUDE.md` 和目录读。

## 20. 重构后验收标准

重构完成后至少能：

1. 无最近登录用户时强制弹出登录弹窗。
2. 通过设置退出账号，并在回到主界面后再次要求登录。
3. 在设置中切换中文/英文，且 UI 不出现中英文并列。
4. 在设置中配置 API，不再出现 AI 启用复选框。
5. 在空工作区状态下显示灰色分级占位节点。
6. 从左上角导入包含唯一标记的工作区，缺失或多个标记时失败。
7. 从灰色主节点 `+` 创建主节点和工作区，并自动生成主节点文件夹。
8. 程序自动分配节点 ID，创建弹窗不要求用户填写 ID。
9. 主界面只有左侧节点/文件树和右侧详情区。
10. 选中任意节点时右侧只显示该节点详情。
11. 保存按钮只在内容发生修改后启用。
12. 图片库在左侧树中与模块同级显示，并支持点击、上传、拖拽、粘贴。
13. 节点详情中的图片关联使用预览图和图片名选择，不使用手填 ID。
14. 每个节点详情中可通过弹窗打开 MD 预览。
15. AI 增加、修改、润色都在节点基础信息下方的“AI辅助编辑”分组框进入弹窗，多字段选择、tag、生成、对比、编辑、确认覆盖流程完整。
16. 模块和游戏汇总融入同一个“AI辅助编辑”分组框，并使用同样的对比确认流程。
17. 左侧右键可粘贴图片、在文件夹中打开选中文件，并可从右键菜单底部删除当前指向的文件、图片或节点。
18. 删除节点或图片时执行权限检查和二次确认。
19. `AGENTS.md` / `CLAUDE.md` 只在用户主动导出时生成或更新。
20. `manifest.yml`、`INDEX.md`、`image_catalog.yml` 只在用户主动导出当前目录时生成或更新。
21. 左侧显示的节点、图片、Agent 文件和索引文件与实际文件层级一致，选中文件时右侧详情显示文件内容预览。
22. 原先冗余 UI 文案和区域全部移除。
